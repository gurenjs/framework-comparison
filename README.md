# Framework Comparison: the same app, three ways

The same minimal multi-user blog — auth, posts CRUD with pagination and
ownership checks, comments, server-side validation, a deferred welcome
notification, and integration tests — implemented three times from the same
[specification](./SPEC.md):

| Directory | Stack |
|-----------|-------|
| [`guren/`](./guren) | [Guren](https://github.com/gurenjs/guren) — fullstack TypeScript framework on Bun (Hono + Drizzle + Inertia/React) |
| [`hono/`](./hono) | [Hono](https://hono.dev) + Drizzle, hand-rolled session auth, Vite React SPA + JSON API |
| [`nextjs/`](./nextjs) | [Next.js](https://nextjs.org) App Router + Server Actions + Auth.js + Drizzle |

All three use TypeScript (strict), SQLite, Drizzle ORM, and React — so the
differences you see are **framework glue**, not ORM or language differences.

## Reading the numbers

**Handwritten LOC is the headline.** To take a fresh project to the full spec,
you type ~608 lines on Guren versus ~977 on plain Hono (−38%) and ~903 on
Next.js (−33%). The rest of Guren's code was produced by generators
(`create-guren-app`, `make:feature`, `make:job`) whose pristine output is
committed under [`baselines/`](./baselines) so the diff is verifiable.

**Raw Source LOC tells the opposite story — deliberately shown.** Guren's tree
is the *largest* (1,032 lines across 39 files), because scaffolds generate
real, owned code: MVC layering, SSR entry points, providers, config. You
maintain more code than you wrote. Whether that trade is good is exactly what
this repository lets you judge.

**Test LOC shows API leverage.** The same SPEC §6 scenarios take 136 lines
with Guren's `TestApp` fluent client, 304 hand-rolled on Hono, and 526 on
Next.js (where integration-testing Server Actions requires notable setup).

**Context tokens measure the full-read cost for an AI agent** — and reading
everything is how an agent must approach the Hono and Next.js apps. Guren
ships a cheaper path: `bunx guren context` emits a project map (routes,
models, controllers, pages) that measures **404 tokens** against ~10,000 for
reading the tree — and `bunx guren check` / `bunx guren audit` verify
route↔controller↔page consistency and validation/auth coverage mechanically
after an agent edits. The other two stacks have no equivalent; verification is
reading the diff.

## Fairness

Each implementation is written idiomatically, the way its framework's own
documentation recommends, and each has its own README documenting setup, tests,
and any SPEC deviations. If you think an implementation misrepresents your
framework of choice, **please open a PR** — improving the competitors'
implementations makes this comparison more useful, not less.

Known asymmetries we chose to keep, because they are each framework's default:
the Guren app is server-rendered (Inertia SSR), the Hono app is a client-side
SPA over a JSON API, and the Next.js app uses Server Components + Server
Actions. Test counts differ (11 / 18 / 26 tests) although all cover the same
SPEC §6 scenarios.
